## more semantic description (with shifting detail portrayed as an optimization)
 
insert as a bucket
 
- binary search twice (start & end) to identify the "sublist"
  (the range of keys in the insert list that are destined for this bucket)
- curr = bucket head
- for each key k in the sublist
    - if k belongs in curr 
        - each thread in the tile picks up a key from curr (called the origKeys)
        - tile computes which origKey is the successor of k; call it testKey
        while (k' does not exceed node-max)
            - *check (space in node)- else exit
            - for each key k' in the sublist where k <= k' < testKey
                - insert k' in curr 

            - find next origKey to become testKey' where k' from sublist <= testKey'
            - write back to curr all origKeys between [testKey and testKey'): 
            ( testKey <= write back origKeys  < testKey')
            (write back occurs in one step)
            - testKey = testKey'
        end while
    - else curr = curr.next
 
    
*check (origKeys are preserved, so if we have no more room for new insertions - write back all remaining origKeys to curr and exit)
  - in this case we will go back around the main for loop, a split node will be invoked on curr (if there are still remaining insertions in the sublist) and then we can re-enter the bulk insertion code 


how we efficiently shift all keys to the right
- we don't *actually* repeatedly shift
- we have all threads LIFT keys (one per thread) into their local memories
- when inserting k', we overwrite any key that already exists in that position
- when we have thread write their LIFTED keys back in order (also overwriting)
- in the end, all original keys are preserved and new keys are inserted in order
